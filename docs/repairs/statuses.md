# Статусы и переходы

Статусная модель определяет жизненный цикл ремонта — от приёма устройства до его выдачи клиенту. Каждый статус отражает текущий этап работы над заказом.

<!-- screenshot: диаграмма статусов ремонта с переходами -->

## Концепция статусов

Статусы упорядочивают работу сервисного центра и обеспечивают прозрачность процесса для всех участников — мастеров, приёмщиков и клиентов.

- Каждый ремонт в любой момент времени находится ровно в одном статусе.
- Переход между статусами возможен только по заданным правилам.
- Смена статуса может запускать автоматические действия (см. [Действия Workflow](workflow-actions.md)).

## Стандартные статусы

Система поставляется с набором предустановленных статусов, покрывающих типовой процесс ремонта.

| Статус | Группа | Описание |
| --- | --- | --- |
| Открыт | Новые | Устройство принято в сервис, ожидает диагностики |
| Диагностика | Диагностика | Мастер проводит диагностику неисправности |
| Согласование | На удержании | Ожидание согласования стоимости с клиентом |
| Согласован | В работе | Стоимость согласована, ремонт готов к выполнению |
| Заказать запчасти | На удержании | Необходимо заказать запчасти для ремонта |
| Запчасти заказаны | На удержании | Запчасти заказаны, ожидается поставка |
| Нет запчастей | На удержании | Запчасти отсутствуют |
| Запчасти получены | К выполнению | Запчасти получены, ремонт готов к продолжению |
| В работе | В работе | Ремонт выполняется |
| Готов | Готовые | Ремонт завершён, устройство готово к выдаче |
| Выдан и не оплачен | Готовые | Устройство выдано, оплата не получена |
| Отказ | Отказ | Клиент отказался от ремонта |
| Закрыт | Закрыт (успех) | Ремонт завершён и закрыт (финальный статус) |
| Утилизирован | Закрыт (неудача) | Устройство утилизировано (финальный статус) |

{% note info %}

Набор статусов можно изменить в разделе [Настройки](../settings/). Цвета назначаются не отдельным статусам, а группам — все статусы одной группы отображаются на канбан-доске в одном столбце с единым цветом.

{% endnote %}

## Правила переходов

Правила переходов определяют, из какого статуса в какой можно перевести ремонт. Это предотвращает ошибки и обеспечивает соблюдение технологического процесса.

<!-- screenshot: настройка матрицы переходов между статусами -->

- **Разрешённые переходы** — для каждого статуса задаётся список статусов, в которые возможен переход.
- **Ограничения по ролям** — переход может быть доступен только определённым ролям (настраивается через Workflow-действия).

## Жизненный цикл ремонта

Типичный путь ремонта через статусы выглядит следующим образом.

```
Открыт → Диагностика → Согласование → Согласован → В работе → Готов → Закрыт
```

Возможны ответвления — ожидание запчастей, возврат на диагностику, отказ клиента. Все эти сценарии настраиваются через правила переходов.

{% note warning %}

Обратные переходы (например, из «Готов» обратно в «В работе») рекомендуется ограничивать и использовать только при необходимости. Каждый обратный переход фиксируется в истории заказа.

{% endnote %}

## Комментарий при смене статуса

При переводе ремонта в новый статус можно оставить комментарий, который будет виден в истории заказа.

- Комментарий может быть обязательным для определённых переходов.
- Комментарий автоматически сохраняется в истории изменений.
- Комментарий может включаться в уведомление клиенту.

## История переходов

Система фиксирует все смены статусов с указанием автора, времени и комментария. История доступна в карточке ремонта.
